OpenApply · 试用任务说明（Paid Trial Task） 
说明：这是一个有偿的短期试用任务。我们相信"一个真实的小任务，比十场面试更能
说明问题"。完成它，既能让我们看到你的真实工作方式，也能让你判断这份工作是否
适合你。 
 
一、背景（你需要了解的） 
OpenApply 是一个已上线的国际升学评估产品：学生在网站上完成一份问卷，系统据
此生成一份个性化的升学分析报告。我们目前的技术栈跑在海外平台（Lovable + 
Supabase） ，报告由云端大模型生成。 
下一步，我们希望建立一个知识库检索能力（RAG），让报告的生成能够基于我们自己
整理的、准确的院校与政策资料，而不是仅依赖模型的通用知识。 
这个试用任务，就是这个方向的第一块拼图。 
 
二、任务目标（范围严格限定，只做这一件事） 
用云端 LLM API，做一个能跑通、且检索准确的 RAG demo。 
具体来说，交付一个可运行的小程序 / 服务，它能够： 
1. 接收一批我们提供的院校/政策资料文档（我们会给你一小份样本数据集） ； 
2. 对文档做切分、向量化、入库（用任意你熟悉的向量库均可） ； 
3. 接收一个用户的自然语言提问（例如"英国哪些学校的硕士毕业后有利于回国落
户？"）； 
4. 从知识库中检索出相关内容，交给云端 LLM，生成一个基于检索结果、而非模
型臆想的回答； 
5. 回答中能体现出"这个结论来自哪份/哪段资料"（可追溯） 。 
明确不在本任务范围内的（请勿做） ： 
•   不需要做前端界面（命令行 / 简单接口即可） 
•   不需要本地部署 GPU 或开源模型（本任务只用云端 API）

•   不需要接入我们现有的生产系统（Lovable / Supabase） 
•   不需要处理高并发、用户系统、登录等 
一句话：做一个"问它一个升学问题，它能从我给的资料里查到依据、并给出准确回答
"的最小可用 demo。 别多做，把这一件事做扎实。 
 
三、验收标准（我们会这样判断好坏） 
我们会用一组我们自己准备的测试问题来评估你的 demo，重点看四件事： 
1. 检索准确性：针对我们的测试问题，它检索到的资料是否相关、是否找全了关
键信息（不漏检、不跑偏） 。这是最重要的一项。 
2. 回答忠实度：回答是否基于检索到的资料，而不是模型自己编的（我们会故意
问一些"资料里没有、但模型可能瞎答"的问题，看它会不会诚实地说"资料中未
提及"）。 
3. 可追溯性：回答能否指出依据来自哪段资料。 
4. 代码与工程质量：代码是否清晰、能否按你的说明顺利跑起来、结构是否合
理。 
 
四、交付方式 
• 代码：放在一个私有 Git 仓库（GitHub/Gitee 均可） ，邀请我们查看；或打包
发给我们。 
• 一份简短说明（README）：如何运行、你做了哪些关键技术选择、以及为什么
这么选（例如：为什么用这个切分策略、这个向量库、这个检索方式） 。我们
很看重"你为什么这么做"。 
• 一段简短演示：录屏或视频通话演示均可，展示它跑起来的样子。 
 
五、时间与报酬 
• 时间：自你确认接受任务起，2 周内交付。
